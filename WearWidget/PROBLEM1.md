# Problem Description: Authentic Previews for Wear Widgets in `compose-preview`

## Background

This document describes the challenge of generating authentic previews for Wear
Widgets using the `compose-preview` tool in the `wear-os-samples/WearWidget`
project. Wear Widgets are partial-height, vertically scrolling surfaces that use
Remote Compose to render content on Wear OS devices.

## The Problem

Current `compose-preview` output doesn't match the real device appearance
(missing frame, rounded corners, header). On a real device, a widget is
typically hosted within a specific "frame" provided by the system (e.g., the
carousel), which often places the widget as a rounded rectangle inside a
circular display, sometimes accompanied by a header (app icon and title).

The standard previews generated by `compose-preview` typically show one of two
things:

1. **Full-Screen Fill**: The widget content expands to fill the entire circular
   canvas (if a round device is selected), ignoring the fact that in production
   it might be restricted to a smaller, rounded rectangle.
2. **Missing Context**: Previews lack the surrounding UI context (like the watch
   face boundary and headers) that helps designers judge glanceability and
   layout correctness.

## The Goal

The goal is to make previews generated by `compose-preview` look **basically
identical** to those generated by deploying to a real device and taking a
screenshot.

Fundamentally, the widget is just a Remote Compose document. The difference lies
in the "frame" or container that the renderer applies around that document.

## What We've Tried (and Current Workarounds)

During this exploration, we attempted to simulate this "letterbox" or
"widget-in-container" effect manually in the preview code.

### 1. Manual Layout Simulation

We created a shared preview wrapper `LetterboxWidgetPreview` and a custom
annotation `@WearLetterboxPreview` in `WidgetPreview.kt`. This wrapper manually
constructs:

- An outer `RemoteBox` filling the screen with a black background.
- A header with a dummy icon (emoji) and title.
- An inner `RemoteBox` with specific dimensions (e.g., 190x150) to contain the
  widget content.

### 2. Simulating Rounded Corners

Since Remote Compose does not currently support a `clip` or `cornerRadius`
modifier on `RemoteBox` that works in these previews, we used `magick`
(ImageMagick) to add transparent rounded corners to the background images used
by the widgets.

### Current Limitations of the Workaround

- **No True Clipping**: Content drawn on top of the background image (like a
  vertical color bar filling max height) is not clipped by the image's rounded
  corners and appears sharp at the edges.
- **Manual Maintenance**: Every preview must use the specific wrapper, and sizes
  must be adjusted manually if content overflows.
- **Font Limitations**: Custom fonts are not supported by Remote Compose (they
  fall back to system defaults), making it impossible to replicate styles that
  rely heavily on specific typography (like Art Deco).

## Potential Avenues for Future Work

To solve this problem systematically without manual hacks in every widget file,
future work should explore modifying the rendering pipeline in the
`compose-ai-tools` repository (source code located at
`/usr/local/google/home/stillers/workspace/compose-ai-tools`).

### 1. Enhancing the Renderer

- **Automatic Framing**: Investigate if the renderer can automatically apply a
  "frame" (like the circle + header) when the preview profile is identified as
  `WEAR_WIDGETS`.
- **True Clipping Support**: Add support for corner radius clipping in the
  renderer or `RemoteModifier` so that content is properly clipped to rounded
  rectangles without needing image hacks.
- **Review Existing Solutions**: The user pointed to a patch at
  `https://android-review.git.corp.google.com/changes/platform%2Fframeworks%2Fsupport~4045856/revisions/20/patch?download&raw`
  for examples of how to handle this. This should be investigated if access can
  be obtained.

### 2. Tooling Improvements

- Allow passing custom device frames or overlays via CLI arguments to
  `compose-preview`.

## How to Judge Success

A solution is correct when `compose-preview` can take a standard `@Preview`
annotated function for a Wear Widget and produce an output image that matches
the structure and visual appearance of the reference images in `drawable` (e.g.,
[weather_widget_fullscreen.png](file:///usr/local/google/home/stillers/workspace/wear-os-samples/WearWidget/app/src/main/res/drawable/weather_widget_fullscreen.png))
automatically, without requiring the developer to manually add black circles or
headers in their preview code.
